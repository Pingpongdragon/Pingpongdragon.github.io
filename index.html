Learning notes : continuous QA datasets

发表于 2025-05-14 更新于 2025-05-15 分类于 learning notes Valine：

本篇文章主要是总结一下现有持续学习文本QA对的数据集。主要有以下几个数据集：

对话系统数据集

MultiWOZ数据集和SGD数据集都是用于持续学习对话系统研究的基准数据集。它们各自的特点和用途如下：

MultiWOZ数据集

  • 收集目的：MultiWOZ数据集是一个用于多领域对话系统的基准数据集，旨在促进对多领域任务型对话系统的的研究。
  • 特点：
    • 多领域：涉及多个不同的领域，如餐馆、酒店、景点、火车和出租车等。
    • 多样性：包含各种类型的对话，用户有不同的需求和偏好。
    • 标注丰富：对话标注包括对话行为、领域、槽位和值等信息。
  • 用途：用于训练和评估多领域任务型对话系统，特别是在对话状态跟踪、对话管理策略和自然语言生成等方面。
  • 数据规模：包含约10,000个对话，涵盖多个领域和任务。
  • 挑战：对话的多样性和复杂性要求模型能够理解和处理多种类型的用户请求，并在不同的领域之间进行有效的切换和整合。
    论文链接：https://arxiv.org/abs/1810.00278
    开源地址：https://github.com/budzianowski/multiwoz.git

SGD数据集

  • 收集目的：SGD（Schema-Guided Dialogue）数据集是为了推动对跨领域任务型对话系统的研究而设计的，提供了丰富的对话样本和对应的数据库模式。
  • 特点：
    • 跨领域：包含多个不同领域的服务，如银行、旅游、家居服务等。
    • 对话复杂性：对话涉及复杂的服务交互，包括多个意图和服务的组合。
    • 数据库支持：提供与对话相关的数据库模式和操作，用于支持对话系统的执行。
  • 用途：用于训练和评估跨领域任务型对话系统，特别是在处理复杂对话逻辑、服务交互和数据库查询等方面。
  • 数据规模：包含约20,000个对话，涵盖多个领域和服务。
  • 挑战：对话的复杂性和跨领域的特性要求模型能够理解和处理多种服务的交互逻辑，并能够有效地利用数据库信息来完成任务。
    论文链接：https://arxiv.org/abs/1909.05855
    开源地址：https://github.com/google-research-datasets/dstc8-schema-guided-dialogue.git

对话（Dialogue）格式

  1. 对话（Dialogue）：

    • dialogue_id：对话的唯一标识符。
    • services：一个列表，包含对话中涉及的服务名称。
    • turns：一个列表，包含对话中的每一轮交互。

  2. 轮次（Turn）：

    • speaker：本轮的说话者，取值为 "USER" 或 "SYSTEM"。
    • utterance：自然语言的话语。
    • frames：一个列表，每个元素对应一个服务的注释信息。

  3. 框架（Frame）：

    • service：对应服务的名称，框架中使用的槽位和意图信息来自该服务的模式。
    • slots（非分类槽位）：一个列表，每个元素表示话语中的一个槽位跨度，包含以下字段：
      • slot：槽位的名称。
      • start：槽位值在话语中的起始字符索引。
      • exclusive_end：槽位值在话语中最后一个字符的下一个字符的索引，在 Python 中，utterance[start:exclusive_end] 可以获取槽位值。
  • 在单域数据集中，每一轮只包含一个框架；在多域数据集中，某些轮次可能包含多个框架。

随时间变化的数据集

  • 首先是timeQA 数据集，是由加州大学圣巴巴拉分校(UCSB)的计算机科学系收集开发的，发表在2021年的NeurIPS的一个数据集，专门用于测试模型回答需要时间推理的问题的能力。时间敏感问题的定义
    论文将时间敏感问题定义为：
    • 包含时间指示符（如”在2007年”或”2010年之前”）
    • 修改时间指示符会导致答案变化
    • 需要进行时间推理
      包含约20K个问答对，涵盖5.5K个随时间演变的事实和70种关系
      分为”容易版”和”困难版”两个版本（具有不同的推理难度），如果要做RAG，他是从维基百科中构建的，相应知识库可以搭建为维基百科知识库，容易版和困难版困难需要不同的检索参数和生成参数。

开源地址https://github.com/wenhuchen/Time-Sensitive-QA

  • 其次是streaming QA数据集，这篇文章是由介绍了用于问答模型适应新知识的benchmark。作者出了当前QA评估的一个关键限制：大多数数据集使用静态知识快照（如维基百科），这无法反映我们动态、不断发展的世界，在这个世界中，模型的知识很快就会过时。
    数据集包含

  • 与特定日期相关的人工编写和AI生成的问题

  • 可以从14年时间戳新闻文章（2007-2020）中找到答案的问题

  • 当模型接触到新信息时进行季度评估
    主要发现：

  • 参数化模型可以逐步更新而无需完全重新训练，避免灾难性遗忘（fine-tune）

  • 半参数化模型通过将新文章添加到搜索空间中可以快速适应（rag）

  • 具有过时语言模型的模型表现不如重新训练的语言模型

  • 参数化更新对高频命名实体的问题特别有益

当前的QA数据集缺乏问题和知识来源的时间基础。

StreamingQA通过以下方式解决这一问题：

  • 为问题和知识来源提供时间元数据
  • 建立模拟现实世界信息流的流式设置
  • 包含关于近期事件（在提问后一个月内）和历史事件的问题
  • 支持相对时间表述（如”3个月前”）
  • 相比维基百科，从新闻文章中提供更现实的挑战（冗余、噪音信息）
  • 研究考察了不同模型架构如何适应新知识：
  • 参数化（闭卷）方法，将知识编码在模型参数中
  • 半参数化（开卷）方法，结合检索和生成
    他们的分析表明这些方法是互补的 - 参数化适应有助于处理频繁实体，而基于检索的适应对不太常见的知识效果更好。所定义的任务为流式，即模型需要使用截至特定时间点发布的文章来回答该时间点提出的问题。对于评估集，它分为两个子集：最近子集问题的文章发布日期采样于2019年12月至2020年12月，问题日期在文章发布后30天内；过去子集问题的文章发布日期采样于2007年至2020年12月，问题日期在2020年1月或文章发布日期之后的365天内。训练和验证集的问题日期范围在2007年至2019年12月之间，文章发布日期分布与评估集类似。

Learning notes : Caravan

发表于 2025-05-13 更新于 2025-05-15 分类于 learning notes Valine：

论文总结：CARAVAN——基于标注代理的在线学习系统

研究背景

传统离线训练的机器学习模型在部署后难以适应动态变化的网络环境（流量波动），导致模型性能下降，产生模型漂移的现象。

  • 挑战
    • 标注难题：在线流量缺乏实时性的标注（ground truth）
    • 再训练时间间隔：频繁再训练计算成本高，但间隔过长会降低性能

核心贡献

CARAVAN 是一个面向网络内机器学习的在线学习系统，主要创新点有以下三个：

  • 通过两种标注方法生成弱监督标签
    第一种是低准确率但高速的控制访问列表，生成部分标签；
    第二种是高准确率但低速的大模型，生成规则供短期内快速标注

  • 采用精度代理，用生成标签代替真实标签，计算模型性能的代理指标，从而检测模型偏移量

  • 动态再训练触发，基于精度代理变化自动触发再训练，比持续训练节省 61.3% 的 GPU 时间

标签代理

标签代理旨在解决在线学习中的自动标注问题，在网络流量分析的场景中，缺乏人工标注的真实标签。而标签代理能够对网络流量特征数据进行分析，提取出重要的特征信息。比如在网络入侵检测中，它可以分析流量的持续时间、协议类型、源和目的 IP 地址，对流量进行分类和识别，判断流量是否正常和相应攻击类型。

它通过以下几步生成标签：

  • 首先是收集流量样本：
    标签代理从数据平面设备（如可编程交换机或智能网卡）收集网络流量样本。每个样本包含一组特定的头部字段（称为流量特征）以及部署的机器学习模型对该样本的预测结果。这些样本按时间窗口进行组织，每个窗口包含一定数量的样本。然后进行数据清洗、特征选择和标准化等操作保证输入数据的特征合理。

  • 其次是利用外部关键方法生成相应标签：
    比如启发式规则、访问控制列表、深度学习模型或基础模型（如 GPT-4、Gemini 等），它们有些是低准确率但高速，另一些是高准确率但低速。

    • 第一个 insight：可以利用低准确率知识源生成弱监督的 label 训练模型，然后低准确率知识源仅对那些源 IP 地址在黑名单上的流量进行标记。因为这些 IP 地址已被认为可能是恶意的，所以对它们的标记更具可信度。尽管只标记了一部分数据（如黑名单 IP 流量），但由于网络中传入的数据量很大，最终仍然可以获得足够数量的标记数据来用于模型验证和再训练。

    • 第二个 insight：可以利用高准确性的知识源，转化为规则集或者启发式，在有限的时间内快速对资源进行高效地标记，但是也会过时，必须偶尔更新。

    为了支持新的知识源，用户只需要完成一个名为 label() 的函数，该函数将一个窗口的数据样本作为输入，并返回该窗口上的一组标签作为输出。

数据偏移检测

模型验证会定期检测和评估网络内的模型性能，负责在必要时触发模型（这里的模型是额外的网络中用于流量分析和决策的机器学习模型）的在线训练。其由两个组件组成：

  • 准确性代理
    用于在不依赖真实标签的情况下，评估和监控模型的性能，并决定是否需要触发重新训练的一个代理。它的目的是替代真实准确率，通过比较网络模型预测结果与前一步生成的标签，判断是否出现了数据偏移。使用的是由标签代理生成的标签而不是真实标签。例如，在计算 F1 分数时，它将模型预测的标签与生成的标签进行比较，并根据匹配程度计算出精确度、召回率和最终的 F1 分数。准确性代理为模型的重新训练提供了一个客观的触发条件。当准确性代理的值低于某个设定的阈值，或者出现了明显的下降趋势时，系统会启动重新训练流程，以更新模型并提高其对新数据的适应性。（也可理解为在验证集上的准确率低于某个阈值？对应 insight：即使无法获得真实准确率，也可以通过观察准确率代理的时间趋势变化来发现潜在的模型漂移。）

  • 再训练触发器
    CARAVAN 的重新训练触发机制有三种模式：

    1. 基于窗口的触发（Window-based）

      • 每隔 X 个标记窗口就定期重新训练一次。当 X = 1 时，CARAVAN 会在每个窗口结束时都进行重新训练，这种方式类似于之前的研究。
      • 对于基于窗口的触发机制，验证阶段会跳过使用准确性代理，因为这种触发方式本身不依赖于准确性代理的值。

    2. 基于准确性的触发（Accuracy-based）

      • 如果准确性代理的值在时间尺度上满足某种特定模式，则触发重新训练。例如，用户可以设定一个准确性阈值，如果准确性代理的值持续低于该阈值，就会启动重新训练。
      • 用户应根据具体应用与准确性代理一起定义这种触发机制。虽然准确性代理能够捕捉到有意义的信号（例如 F1 分数下降），但重新训练触发机制明确指定了用户希望在线训练发生的时间，这在不同的应用中可能会有很大差异。

    3. 基于事件的触发（Event-based）

      • 当发生特定事件时触发重新训练，例如当标记代理或人工操作员检测到某种特定类型的攻击时。

在 CARAVAN 中，主要关注基于准确性的重新训练触发机制，即利用准确性代理的值来决定是否需要进行在线训练。重新训练触发机制需要做出两种决策：

  • 如果在上一个标记窗口结束时没有进行重新训练，那么是否应该在这个窗口进行重新训练？CARAVAN 的重新训练触发机制会比较当前窗口和上一个窗口的准确性代理值，如果观察到当前窗口的准确性代理值出现急剧下降，则会启动重新训练，因为这可能是潜在的概念漂移或数据漂移的明确信号。
  • 如果在上一个标记窗口结束时已经进行了重新训练，那么是否应该停止在这个窗口继续重新训练？如果连续对多个窗口进行重新训练，在没有新的漂移出现的情况下，边际推理准确率增益会逐渐减少。因此，如果最近几个窗口都已经进行了重新训练并且获得了不错的推理准确率增益，那么重新训练触发机制就会停止重新训练。
